### 📌 12월 27일 (1) - 아토믹 디자인을 최대한 잘 적용해보았다

이 프로젝트는 아토믹 디자인으로 구성되어 있다. 페이지를 컴포넌트 단위로 나누어 관리하는 리액트의 특성상 아토믹 디자인을 적용하면 컴포넌트의 재사용성을 끌어올리고 유지보수 측면에서도 훌륭할 것이라 생각했기 때문이다. 가장 기본이 되는 atoms 컴포넌트들, atoms 컴포넌트들이 모여 한 단계 상위 단계의 molecules, 그 다음 단계로 organisms 로 구성하고 templates과 pages는 개츠비 프로젝트 특성상 components 디렉토리 밖에 생성하여 관리하고 있다. 아토믹 디자인은 컴포넌트들을 다음 단계로 분리한다.
<br/>
<br/>
atoms —> molecules —> organisms —> templates --> pages
<br/>
<br/>
단계가 올라가면서 컴포넌트들의 재사용성은 줄어들고 당연히 pages는 단 한번만 쓰인다. 직접 적용해보니 organisms 단계만 되더라도 거의 재사용하지 않는다는 것을 느꼈다. 프로젝트가 진행될수록 프로젝트의 크기 또한 커질수밖에 없는데 지금까지의 개발 과정에서만 바라보아도 아토믹 디자인은 정말 괜찮아 보인다. 재사용이 가장 많이 되는 컴포넌트들이 위치적으로도 모여있으니 유지보수 측면에서도 그 장점이 곧바로 체감되었고 이후 개발 과정에서도 모듈들을 관리하기에 편할 것이라는 생각이 들었다.
<br/>
<br/>
하지만 어디까지가 각 단계인가를 규정하는 것에 있어서 애매하다는 생각이 들었다. 아토믹 디자인의 표준이 정해져 있는지는 아직은 잘 모르겠지만 지금은 개인으로 프로젝트를 진행하고 있기 때문에 각 단위의 정의가 확실하지 않아도 상관없이 진행할수 있어 일단 개발을 진행하였다. (개발 후반부에 다시 알아보며 그때 모듈을 나누어도 괜찮은 공부 방법인거 같다) 그래도 팀 프로젝트에서는 반드시 이에 대한 정의를 확실히 해두는 편이 좋을 것이라 생각이 든다. (이 프로젝트는 Header를 organism이라 정의하고 이와 비슷한 단위들을 organism이라 정의하였다.)
<br/>
<br/>
디자인을 적용하더라도 결합도를 낮추고 응집도를 올리기 위한 노력이 필요하다. 직전 학기에 배운 소프트웨어 공학 과목에서 결합도를 낮추고 응집력을 올려야 모듈의 재사용성을 올릴 수 있다라고 배웠다. 아토믹 디자인을 잘 적용해야 확장성과 재사용성을 보장해줄 수 있다. 이에 대한 노력을 해야하고 충분한 고민이 필요하다. 특히나 앞으로 개발을 하면서 atoms에 있는 컴포넌트들은 적재적소에 쓰여야 할 것이다.

### 📌 1월 3일 - 다크 모드 구현중.... 브라우저가 렌더링하는 과정을 자세히 알 필요가 있을 듯 하다.
- 상태관리 라이브러리는 recoil을 썼다.
- develop에서는 문제 없던 것이 build시에  window is not defined. 에러가 난다.
- 그 이유는 서버 사이드에서 실행되던 develop은 문제가 없지만 nodejs 환경인 build time에는 
- window와 같은 브라우저 전역의 API를 사용할 수 없다는 것...
- typeof window !== 'undefined' 를 통해 에러 자체는 해결할 수 있었지만
- 다크 모드 자체에는 심각한 오류가 있었다.
- 내가 구현한 다크모드의 디폴트는 localstorage, 없다면 system mode 인데
- localstorage가 dark라고 설정되어 있을 때, 첫 렌더링에 바로 다크 모드가 되지 않고 페이지를 옮기면 적용이 된다.
- 즉 build 시점에 localstorage에 대한 정보가 없으니 발생하는 문제인 것이다.
- 이 부분은 gatsby가 지원하는 gatsby-ssr.js를 통해 해결할 수 있다고 한다.
